# PRD: Sonnet 4.6 Upgrade for Brain Atom Generation

**Date:** 2026-04-21
**Author:** Rob Gabel + Claude (follow-up to the Opus 4.7 synthesis upgrade pass, commits `ca9f892` + `681254f`)
**Status:** Deferred — execute when audit scores drop, user feedback highlights atom quality, or a new model tier makes the delta larger
**Priority:** P2 — current brains score 97/100 avg; the upgrade sharpens but doesn't rescue

---

## The Problem

The 15 live brains were built in phases. Their content comes from two generations of models:

- **Synthesis (Phase 3):** Upgraded to Opus 4.7 on 2026-04-21. All 15 brains now have synthesis generated by the current frontier model.
- **Atom generation (Phase 2):** Still on Sonnet 4 (`claude-sonnet-4-20250514`, the May 2025 snapshot) for every brain except those built after the Phase 2 model was updated in `scripts/auto_build_config.py`. Per git log, only Oprah Winfrey was built start-to-finish with Sonnet 4.6 (commit `2d5e27b`). The other 14 brains' atoms — content, original_quote, implication fields — were distilled by Sonnet 4.

Phase 2 is the bulk of the LLM work per brain: ~200–400 atoms generated across 12–16 clusters, ~1M output tokens. This is where voice preservation in the `content` field, choice of `original_quote`, and sharpness of `implication` happen. Sonnet 4 → 4.6 is ~10 months of model progress compressed into this single phase.

**What a smarter Sonnet would plausibly change:**
- Tighter distillations in the `content` field (less corporate-speak paraphrasing)
- Better `original_quote` selection (picking the most revealing quote from a long transcript, not the first one that scans well)
- More precise `implication` fields (the "so what" — currently generic on lower-tier atoms)
- Sharper cluster taxonomy in Phase 1 (brain.json scaffolding) — if Phase 1 also reruns
- Fewer generic atoms that score low on confidence

**What it won't change:**
- Atom count (driven by prompt structure + cluster count, not model)
- Source coverage (bounded by `research/` and `source/` directories)
- Connection graph density (topic overlap is deterministic; Haiku 4.5 does typing and is unchanged)
- Voice enrichment fields (Haiku 4.5 unchanged)

**Why this is deferred, not urgent:**
- Current audit scores average **97/100** across the 15 live brains. Most are already at 99–100.
- Spot-checks of synthesis output (post-Opus 4.7) show the content is already strong and specific.
- The Opus 4.7 synthesis pass only cost ~$5 total across 4 brains. A full Sonnet 4.6 Phase 2 rerun is **~$20 per brain** and has Supabase collision risks that need engineering around. Payoff/cost ratio is much worse than the synthesis upgrade.
- Better signal to upgrade: user feedback on specific brains feeling generic, OR a new model tier that moves the delta into clearly-worth-it territory.

## Approach

### The Hard Part: Supabase Collision

Phase 2 ends with `httpx.post(f"{url}/rest/v1/{table}", ..., json=batch)` — plain INSERT, not UPSERT. Re-running Phase 2 on an existing brain **duplicates every atom row** in `<slug>_atoms`, and Phase 4 enrichment then runs on the merged v1+v2 set, producing a hybrid graph. Export-brain.py reads from Supabase and inherits the duplicates.

Three viable resolutions, in order of preference:

#### Option A: Truncate + rebuild (recommended)
1. Export current `<slug>_atoms` and `<slug>_connections` tables to local JSON backups (v1 snapshot).
2. `TRUNCATE <slug>_atoms, <slug>_connections` (cascade deletes enrichment rows).
3. Run `auto-build-brain.py --person "<Name>" --skip-phase 0 --resume-from 2` (uses existing source/research, skips source rediscovery, reruns Phase 2+ with current config = Sonnet 4.6 / Opus 4.7 / Haiku 4.5).
4. Re-export pack, re-audit.

Pros: clean v2, clean Supabase, comparable v1 backup retained locally.
Cons: destructive on Supabase (backed up first, so recoverable).

#### Option B: Parallel slug (safest but most overhead)
1. `cp -r brains/peter-attia brains/peter-attia-v2`.
2. Create `peter_attia_v2_atoms` and `peter_attia_v2_connections` Supabase tables.
3. Run `auto-build-brain.py --person "Peter Attia" --slug peter-attia-v2 --skip-phase 0`.
4. Diff pack files side-by-side.
5. If v2 wins, promote: rename tables, swap directories, update `brains/index.json`.

Pros: no data destruction, clean A/B.
Cons: doubles Supabase tables during the transition, requires table swapping logic, more moving parts.

#### Option C: Pipeline code change (proper long-term fix)
Modify Phase 2 upload to use `.upsert()` with `on_conflict="content,source_ref"` (or an explicit natural key). Then re-running Phase 2 updates in place instead of duplicating.

Pros: enables easy reruns forever.
Cons: requires deciding what "same atom" means (content hash? source + cluster?). Risk of silently dropping legitimately different atoms. Needs design + test.

**Recommendation:** Start with Option A on a single-brain pilot. If it works cleanly and the delta is worth it, use Option A for the rollout. File Option C as a follow-up improvement once we've learned from the pilot.

### Phase-by-phase execution

For each brain being upgraded, the flow is:

| Phase | Action | Model | Why |
|---|---|---|---|
| Backup | Export Supabase tables to `brains/<slug>/backups/v1-{atoms,connections}.json` | — | Rollback window |
| 0 | Skip (`--skip-phase 0`) | — | Don't rediscover sources; we want the model delta, not a content refresh |
| 1 | Run | Sonnet 4.6 | Regenerate cluster taxonomy with the new model. Optional — can skip with `--skip-phase 1` to hold clusters constant (cleaner A/B) |
| 2 | Run (after TRUNCATE) | Sonnet 4.6 | The actual upgrade — regenerate atoms |
| 3 | Run (clear `brain.json["synthesis"]` first) | Opus 4.7 | New atoms → new synthesis. Use the `--force-synthesis` workaround until that flag exists (see `IMPROVEMENTS.md` → Architecture & Distribution) |
| 4 | Run | Haiku 4.5 | Re-enrich voice + connections (atom IDs changed) |
| 5 | Run | — | Export pack + audit |

### Pilot-first: pick a brain where the delta is knowable

Same logic as the Opus 4.7 pilot: pick a brain where the current atoms feel generic OR where the source material is noisy (podcasts, tweets) so Sonnet 4.6 has room to improve. Candidates from the audit:

- **Peter Attia (87/100)** — lowest score, 153 atoms (below 250 target), 76% voice, podcast-heavy transcripts. Good A/B candidate.
- **Elon Musk (99/100)** — tweets are low-signal inputs; Sonnet 4.6 might pick better quotes.
- **Steve Jobs (97/100)** — historical, 170 atoms, text-rich interviews. Room to improve.

Don't pilot on brains at 100/100 — there's no headroom to measure.

## Phased Rollout

### Phase 1: Pilot (1 brain, $25, ~90 min)
- Pick Peter Attia (lowest score, noisiest source, most headroom).
- Run Option A: backup → truncate → full rebuild via `auto-build-brain.py --skip-phase 0`.
- Compare v1 vs v2 on:
  - 20 random atoms side-by-side (content sharpness, quote selection, implication quality)
  - Cluster taxonomy diff (if Phase 1 reran)
  - Audit score delta
  - A `/advise` test on a Peter Attia question — does the answer feel more like him?
- **Gate:** If atom-level diff shows clear improvement on 60%+ of sampled atoms AND audit score holds or improves → proceed to Phase 2. If not → stop, file the null result in `IMPROVEMENTS.md`.

### Phase 2: Selective rollout (3–5 brains, ~$100, ~8 hours)
- Target brains where Phase 1 evidence suggests the largest gain:
  - Brains scoring <95 on audit
  - Brains with noisy sources (podcast/video-heavy: Peter Attia, Peter Zeihan, Elon Musk, Gary Vee)
  - Brains where users have given feedback about atom quality (if any)
- Run Option A on each.
- Retest /advise, /teach, /surprise on each for sanity.

### Phase 3: Full cohort (defer decision)
- Only run if Phase 2 showed consistent clear wins AND product signal (user feedback, conversion) justifies the ~$230 and ~15 hours.
- Likely NOT worth doing for the already-100 brains (Munger, John Green, Gary Vee, Jensen Huang).

## Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Supabase duplication breaks pack export | High (default behavior) | Data integrity | Option A truncates before rebuild; backup first |
| Sonnet 4.6 output is no better than Sonnet 4 | Medium | $25 + 90 min wasted on pilot | Pilot first; gate on evidence before rolling out |
| Atom count drops below 250 target | Low | Audit score loss | Pipeline has supplemental research step for thin clusters — already handled |
| Connections lost after truncate | N/A | No impact — re-enrichment regenerates | Phase 4 runs automatically, reproduces graph |
| Voice enrichment quality degrades | Low | Quality regression | Haiku 4.5 is the same model — same output quality expected. Spot-check post-rebuild |
| Cluster taxonomy changes and breaks existing user bookmarks / skill examples | Medium if Phase 1 reruns | Skill example drift | Use `--skip-phase 1` to hold clusters fixed; regenerate only atoms |
| Oprah Winfrey-style build collision — concurrent session commits incomplete state | Low | Confusing git history | Run pilots in one focused session; commit atomically |

## Decision Gates

**Gate 1 (before pilot):** Is there evidence worth spending $25 on?
- [ ] User feedback or spot-check has identified an atom-quality problem
- [ ] OR a new model tier (Sonnet 4.7, 5.x) has been released and the delta jumps
- [ ] OR audit scores have drifted downward for reasons Phase 3 alone can't fix
- If none: defer further. Current state is 97/100 avg — good enough.

**Gate 2 (after pilot, before rollout):** Is the delta worth rolling out?
- [ ] 60%+ of sampled atoms show clearer/sharper content field
- [ ] Voice preservation holds (original_quote still reflects actual phrasing)
- [ ] Audit score unchanged or improved
- [ ] `/advise` output feels more thinker-specific on a blind comparison
- If fewer than 2 of these are true: file as null result and stop.

**Gate 3 (after Phase 2, before full cohort):** Does product signal justify $230 more?
- [ ] Real users are actively using the upgraded brains
- [ ] Feedback or telemetry shows the upgraded brains outperform
- [ ] There's a concrete product moment (launch, feature, partner) that benefits from "every brain on the newest model"
- Default: skip Phase 3. New brains built from here on use the current config automatically.

## Verification

For each upgraded brain:

1. **Supabase integrity check** — row count matches new pack atom count, no duplicates:
   ```sql
   SELECT COUNT(*) FROM <slug>_atoms;  -- should match pack count
   SELECT content, COUNT(*) FROM <slug>_atoms GROUP BY content HAVING COUNT(*) > 1;  -- should be empty
   ```
2. **Audit score** — `python3 scripts/audit-brains.py --brain <slug>` — expect ≥ v1 score.
3. **Pack file sanity** — `pack/brain-atoms.json` has: atoms[], connections[], topic_index, synthesis section populated. `pack/brain-context.md` renders cleanly. `pack/explore.html` loads and shows tabs.
4. **Skill smoke test:**
   ```
   /brain <slug>
   /advise <something the thinker has strong views on>
   /surprise
   /teach <a concept they'd explain>
   ```
   Compare output feel vs v1 (save v1 outputs before rebuild).
5. **brainsforfree.com** — check `website/public/brains/<slug>/` files are in sync (export-brain.py copies them automatically). Visit the live site and confirm the brain page renders.

## Known Unknowns

- Does Sonnet 4.6 actually produce sharper atoms for this specific prompt shape? Unknown until pilot. The Opus 4 → 4.7 upgrade produced 45%+ sharper synthesis for brains with thin drafts, but near-identical output for brains with populated drafts. Phase 2 atom generation doesn't have the same "draft anchoring" dynamic (each atom is generated per-cluster from raw source text), so the delta should be more consistent — but unverified.
- How much does cluster redesign (Phase 1 with Sonnet 4.6) move the needle vs. skipping Phase 1? Skipping is cleaner for A/B; running is closer to a true fresh build.
- Is there a natural key we could use to UPSERT atoms instead of INSERT? (Option C viability.) Probably `(content_hash, source_ref)`, but needs design.
- Does the `/board` skill performance change meaningfully with sharper atoms across multiple brains? Cross-brain quality might be the real payoff.
